什么是merge

情景模拟

假设我们从C1处创建了dev分支，并提交了C2、C3。假设此时别的同事已经向master分支上提交了C4、C5。其示意图如下所示：

我们可以创建一个文件夹，并使用如下命令模拟该过程：

rm -rf .git
git init
# 创建README.md文件
rm README.md
touch README.md
# 向README.md中写入一行内容并提交
git checkout master
echo "C1" >> README.md
git add README.md
git commit -m "C1"

git checkout -b dev
echo "C2" >> README.md
git add README.md
git commit -m "C2"

echo "C3" >> README.md
git add README.md
git commit -m "C3"

git checkout master


echo "C4" >> README.md
git add README.md
git commit -m "C4"


echo "C5" >> README.md
git add README.md
git commit -m "C5"

问题

在上述情境下，我们应该如何将dev上的提交（C2、C3）合并到master？

一般情况下我们会使用merge来合并。具体命令如下所示：（这里忽略了处理冲突的步骤）

git checkout master
git merge dev

合并之后的分支示意图如下所示：

可以看到，Git新创建了一个提交C6，C6同时以C3和C5作为父节点。

合并分支时应该选择rebase还是merge？

当开发完成时，我们需要首先将自己的commits rebase到origin/master上，之后再向主分支提交修改。

假设开发完成后的分支如下图所示：

我们可以首先使用如下命令将dev分支rebase到master分支的最新提交上：

git checkout dev
git rebase master

rebase之后的分支如下图所示：

之后我们可以再次执行merge命令，更新master分支：

git checkout master
git merge dev

更新之后的分支示意图如下所示：

rebase之后再merge的优点

  1. 我们在rebase时处理可能发生的冲突，项目的维护者就不需要处理冲突，直接合并即可。
  2. 由下面的对比可知，rebase之后的分支是线性的，而merge后的提交是图状的。
  1. 如果别人可能基于某些提交进行开发，那么不要执行rebase。（例如，不要在master分支上执行rebase，只应该在我们私有的分支执行rebase。）
    1. 因为rebase的本质是丢弃一些现有的提交，然后新建一些内容相同但实际上不同的提交。

什么是reset

reset就是重置当前的head到指定的状态，其中soft、mixed、hard的区别如下所示：

我们一般会使用reset来撤回最近一次提交。

如何删除某一次提交

删除最近一次提交

删除最近一次提交我们可以使用reset命令

  • 如果我们想将该提交的修改保存在暂存区，我们可以使用—soft ；
  • 如果我们想保留该提交的修改，但不想保存在暂存区，那么我们可以使用—mixed；
  • 如果我们想要完全丢弃该提交的修改，那么我们可以使用—hard。

删除历史提交

假设我们要删除倒数第2次提交C5（这里也可以使用commit id），我们可以首先执行如下命令：

（rebase的时候，参数最少应该是要删除提交的前一个提交。这里参数即为倒数第三次提交）

git rebase -i HEAD~3

此时得到的页面如下所示：

pick b9ee246 C5
pick 4d0ef58 C2
pick 805822f C3

我们将C2前的pick修改为s

pick b9ee246 C5
s 4d0ef58 C2
pick 805822f C3

之后即可删除倒数第二次提交。（rebase本质是以C5为起始点，重新执行一遍C2之后的提交，这里即为C3。）

什么是cherry-pick

该方法会将一条或多条commit应用到当前分支。

假设代码仓库有master和feature两个分支：

a - b - c - d   Master
         \
           e - f - g Feature

现在我们执行如下指令：

# 切换到 master 分支
$ git checkout master

# Cherry pick 操作
$ git cherry-pick f

之后代码仓库将变成这个样子

a - b - c - d - f   Master
         \
           e - f - g Feature
