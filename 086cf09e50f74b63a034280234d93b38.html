什么是merge

情景模拟

假设我们从C1处创建了dev分支，并提交了C2、C3。假设此时别的同事已经向master分支上提交了C4、C5。其示意图如下所示：

我们可以创建一个文件夹，并使用如下命令模拟该过程：

rm -rf .git
git init
# 创建README.md文件
rm README.md
touch README.md
# 向README.md中写入一行内容并提交
git checkout master
echo "C1" >> README.md
git add README.md
git commit -m "C1"

git checkout -b dev
echo "C2" >> README.md
git add README.md
git commit -m "C2"

echo "C3" >> README.md
git add README.md
git commit -m "C3"

git checkout master


echo "C4" >> README.md
git add README.md
git commit -m "C4"


echo "C5" >> README.md
git add README.md
git commit -m "C5"

问题

在上述情境下，我们应该如何将dev上的提交（C2、C3）合并到master？

一般情况下我们会使用merge来合并。具体命令如下所示：（这里忽略了处理冲突的步骤）

git checkout master
git merge dev

合并之后的分支示意图如下所示：

可以看到，Git新创建了一个提交C6，C6同时以C3和C5作为父节点。

什么是rebase

情景模拟

情景模拟同上：

之后，假设我们执行如下指令：

git checkout dev
git rebase master

rebase之后的分支示意图如下所示:

可以看到，相当于dev分支上的两个提交C2、C3被迁移到了master分支所指向的节点上。（实际上是Git以C5为起始分支重新执行了一遍后序的修改（C2、C3））

什么时候使用rebase？什么时候使用merge？

  1. 当开发完成时，我们需要首先将自己的commits rebase到origin/master上，之后再向主分支提交修改。这样，项目的维护者就不需要处理冲突，直接合并即可。

假设开发完成后的分支如下图所示：

我们可以首先使用如下命令将dev分支rebase到master分支的最新提交上：

git checkout dev
git rebase master
